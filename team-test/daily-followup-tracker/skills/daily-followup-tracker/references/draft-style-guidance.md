# Draft style guidance

How drafting agents apply the user's voice profile to follow-up drafts. Each finding's draft is generated by the sub-agent that detected it (email-scan drafts the email replies, per-meeting-scan drafts the meeting follow-ups, etc.). This file is the shared style reference all of them use.

## How drafting agents use the voice profile

Every draft is keyed to a destination channel. The sub-agent picks the section of the voice profile that matches:

- **Email destination** → read `## Email voice` from the voice profile.
- **Slack destination** → read `## Slack voice` from the voice profile.
- **Meeting follow-up** → the destination is determined by `meeting_followup_default_channel` from the wiring config. If it's `email`, use `## Email voice`. If it's `slack`, use `## Slack voice`. This is a one-time choice the user made at setup; never re-decide per finding.

Apply the voice patterns in order: greeting → body → signoff → signature. Match the user's length conventions for the channel — emails tend to be paragraph-shaped; Slack tends to be one or two lines. A wall-of-text Slack draft fails the channel even if every word is in voice; a one-line email draft feels curt even if the body would fit on Slack. Channel shape matters as much as word choice.

## When voice_profile_missing is true

If the orchestrator passed `voice_profile_missing: true`, the user has no captured voice profile (or the file is missing). The drafting agent still produces drafts so the report is useful — but with three guardrails:

- Use a neutral, professional default tone — clear, courteous, no slang, no idiosyncratic phrases.
- Prepend `[VERIFY VOICE]` as a flag at the top of each draft. The user knows at a glance that the draft hasn't been styled to their voice.
- The report header carries a banner reminding the user to re-run setup so future runs use their actual voice. The drafting agent doesn't generate the banner — the orchestrator handles that — but the agent should still flag each draft.

## Anti-patterns

What drafts must avoid. Each one has produced a draft the user would not send.

- **Don't reference being an AI or describe the detection process.** "I noticed in your email that you mentioned..." — no. The draft is written *as the user*, in first person. The user sends it; the AI is invisible in the output.
- **Don't paste verbatim chunks of the original message, transcript, or task description into the draft.** Paraphrase. A draft that quotes the inbound message back at the recipient sounds like a reply written by someone who didn't bother reading the thread.
- **Don't generate drafts longer than the user's natural style for the channel.** Slack drafts especially — if the user's Slack voice runs to one or two short sentences, don't produce a five-line message just because the source content was long. Match the length of the user's typical messages on that channel.
- **Don't apply email voice to a Slack draft or vice versa.** The two channels have genuinely different idioms — Slack tolerates lowercase, contractions, no greeting; email expects a greeting and a signoff. Crossing the channels produces drafts that read as off even when the words are right.

## Draft templates by finding type

Templates below are starting shapes. The actual draft content fills in from the finding's data and the user's voice profile.

- **Open loop reply (email)** — full email shape: greeting (matched to voice — "Hi <name>", "<name>,", or whatever the user uses) + brief acknowledgment of the wait + the answer or next step the user owes + signoff + signature. Keep it tight; the user is replying after a delay, not writing an essay.

- **Open loop reply (slack)** — short. Lowercase if the user's Slack voice tends that way. No greeting unless the user uses one in DMs. Just the answer or next step. Often two sentences or fewer.

- **Today's-new commitment confirmation (email)** — typically suppressed. The original email already went out; the report just surfaces the commitment so the user remembers it. Generate a draft *only* if the commitment was vague on deadline and the user might want to send a follow-up nailing the date down — and even then, frame it as a quick clarifying note, not a full message.

- **Today's-new commitment (slack)** — usually no draft. The commitment is already in-channel; surfacing it in the report is enough. If the user committed to something with no deadline and might want to lock one in, generate a one-line "btw — aiming for <date> on this" style draft, but err on the side of no draft.

- **Meeting follow-up (channel = email)** — *"Hi <name>, following up on our call earlier — <commitment paraphrased>. <next step or deadline confirm>. <signoff>"*. The paraphrase is short — the recipient was on the call, they don't need a recap. The point is to put the commitment in writing and confirm next step or timing.

- **Meeting follow-up (channel = slack)** — short version of the above. *"hey <name> — quick note from earlier: <commitment paraphrased>. will <next step> by <date>."* — or whatever the user's Slack voice favors. One or two sentences.

- **Task next-action** — *not a draft, a suggestion*. The report surfaces a one-line next action the user could take, written *to the user*, not to a recipient. Examples: *"Send status update on Q3 budget review to project owner"*, *"Mark the deck-review task complete and close out — it's been done since Tuesday"*, *"Ping Sam for the missing dependency on the launch checklist"*. No voice profile involved — these are notes, not messages.
